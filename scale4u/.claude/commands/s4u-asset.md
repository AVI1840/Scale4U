# /s4u-asset - נכס ולידציה

קלט: `opportunities/<id>.yaml` עם G1 שעבר.

תהליך:
1. **בחר סוג נכס** לפי השאלה שצריך לאמת (לא לפי מה שמרשים):
   - clickable-demo: לאימות שמשתמשים מבינים ורוצים את הפתרון
   - simulation: לאימות לוגיקה/חישוב (למשל מנוע כללים על תרחישים)
   - process-model: לאימות תהליך עבודה חדש
   - mockup: כשמספיק מסך סטטי לשיחת ולידציה
   עדכן asset_type בקובץ ונמק בשורה.
2. **ייצר דאטה סינתטי** - `assets/<id>/synthetic-data.json`. אסור דאטה אישי אמיתי. נתונים מציאותיים בערכיהם אך בדויים בזהותם.
3. **בנה את הנכס** ב-`assets/<id>/` - ברירת מחדל: קובץ HTML/React יחיד, עצמאי, נפתח בדפדפן ללא שרת.
   רזה: רק ה-Flow שעונה על שאלת הוולידציה. בלי login, בלי settings, בלי קישוטים.
4. **תווית חובה בראש הנכס (באנר גלוי):**
   "נכס ולידציה - לא מוצר | דאטה סינתטי | מורכבות יישום: <מהקובץ> "
5. **תכנית בדיקה:** עדכן test_plan - מי (5-8 משתמשים אמיתיים), מה השאלה, מה קריטריון הצלחה כמותי.
5b. **צ'קליסט "טוב מספיק" (חובה לפני מסירה לבדיקה) - מונע שונות איכות שתפיל את הדמו:**
   - [ ] הנכס פותר את שאלת הוולידציה המקורית - לא הדגמה כללית
   - [ ] אינטראקטיבי דיו כדי שמשתמש יבצע את הפעולה האמיתית (לא רק יסתכל)
   - [ ] דאטה סינתטי בלבד, אך מציאותי בערכיו
   - [ ] תווית הנכס מוצגת בראש (לא מוצר / סינתטי / מורכבות)
   - [ ] נפתח בדפדפן ללא שרת, ללא תלות חיצונית
   - [ ] רזה: אין מסכי login/settings/קישוט שלא משרתים את שאלת הוולידציה
6. הנחה את המשתמש: הרץ את הבדיקה ביום 4, חזור עם ממצאים, נריץ /s4u-decide.

אחרי הבדיקה: תעד findings בקובץ כציטוטים ותצפיות - לא פרשנות - וקבע validation.verdict.
